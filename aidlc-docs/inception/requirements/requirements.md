# ブックマーク管理システム - 要件ドキュメント

**プロジェクト名**: ブックマーク管理システム  
**プロジェクトタイプ**: Greenfield（新規開発）  
**作成日**: 2026-03-01  
**バージョン**: 1.0

---

## 1. 概要

### 1.1 プロジェクト概要
AWS Cognito 2要素認証を使用したブックマーク管理システムの開発。ユーザーはブックマークの一覧表示、詳細表示、編集、削除、新規登録が可能。

### 1.2 要件分析サマリー

**要求タイプ**: 新規プロジェクト（Greenfield）  
**初期スコープ**: マルチコンポーネント（フロントエンド、バックエンド、認証、データベース）  
**複雑度**: 中程度～複雑（AWS Cognito統合、CRUD操作、サーバーレスアーキテクチャ）  
**タイムライン**: 1-2週間（MVP/プロトタイプ）  
**予算制約**: 厳しい予算 - AWSコストを最小化、可能な限り無料枠を使用

## 2. 機能要件

### 2.1 コア機能

#### 2.1.1 ブックマーク管理（CRUD操作）

**FR-001: ブックマーク一覧表示**
- **説明**: ユーザーは自分のブックマークを一覧で表示できる
- **表示方式**: フラットリスト形式（整理機能なし）
- **表示項目**: URL、タイトル、説明、タグ、カテゴリ、メモ、サムネイル、作成日時、更新日時
- **優先度**: 高

**FR-002: ブックマーク詳細表示**
- **説明**: ユーザーは個別のブックマークの詳細情報を表示できる
- **表示項目**: すべての保存されたフィールド（URL、タイトル、説明、タグ、カテゴリ、メモ、サムネイル、作成日時、更新日時）
- **優先度**: 高

**FR-003: ブックマーク新規登録**
- **説明**: ユーザーは新しいブックマークを登録できる
- **入力項目**:
  - 必須: URL、タイトル
  - 任意: 説明、タグ、カテゴリ、メモ、サムネイル
- **自動設定**: 作成日時、更新日時
- **優先度**: 高

**FR-004: ブックマーク編集**
- **説明**: ユーザーは既存のブックマークを編集できる
- **編集可能項目**: URL、タイトル、説明、タグ、カテゴリ、メモ、サムネイル
- **自動更新**: 更新日時
- **優先度**: 高

**FR-005: ブックマーク削除**
- **説明**: ユーザーは不要なブックマークを削除できる
- **確認**: 削除前に確認ダイアログを表示
- **優先度**: 高

#### 2.1.2 検索・フィルター機能

**FR-006: 基本検索機能（初期バージョン）**
- **説明**: タイトルおよびURLによる基本的なテキスト検索
- **検索対象**: タイトル、URL
- **検索方式**: 部分一致検索
- **優先度**: 高（初期実装）

**FR-007: 高度な検索・フィルター機能（将来拡張）**
- **説明**: タグ、カテゴリ、日付範囲によるフィルター機能
- **フィルター項目**:
  - タグでフィルター
  - カテゴリでフィルター  
  - 日付範囲で絞り込み
- **実装時期**: 将来のバージョン（段階的実装）
- **優先度**: 中（将来機能）

**注記**: タグおよびカテゴリフィールドはデータとして保存されるが、初期バージョンではUIでの整理機能（フィルタリング、グループ化）は提供しない

#### 2.1.3 認証機能

**FR-008: ユーザー認証（AWS Cognito）**
- **説明**: AWS Cognitoを使用したユーザー認証
- **認証方式**: ユーザー名/メールアドレス + パスワード
- **優先度**: 高

**FR-009: 2要素認証（MFA）**
- **説明**: セキュリティ強化のための多要素認証
- **MFAオプション**: ユーザーが以下から選択可能
  - SMSテキストメッセージ
  - 時間ベースのワンタイムパスワード（TOTP）認証アプリ
  - メール確認コード
- **優先度**: 高

**FR-010: ユーザー管理**
- **説明**: 基本的なユーザー管理機能
- **ユーザータイプ**: 単一タイプ（すべてのユーザーが同じ権限）
- **機能**: サインアップ、ログイン、ログアウト、パスワードリセット
- **優先度**: 高

### 2.2 データ構造

**BR-001: ブックマークデータモデル**

包括的なデータ構造を持つが、UIでは段階的に機能を実装：

| フィールド名 | データ型 | 必須/任意 | 説明 | 初期UI対応 |
|------------|---------|----------|------|-----------|
| id | String (UUID) | 必須 | ブックマークの一意識別子 | ○ |
| userId | String | 必須 | 所有者のユーザーID | ○ |
| url | String | 必須 | ブックマークURL | ○ |
| title | String | 必須 | ブックマークタイトル | ○ |
| description | String | 任意 | ブックマークの説明 | ○ |
| tags | Array[String] | 任意 | タグのリスト | ○（保存のみ、フィルター機能は将来） |
| category | String | 任意 | カテゴリ | ○（保存のみ、フィルター機能は将来） |
| notes | String | 任意 | メモ | ○ |
| thumbnail | String (URL) | 任意 | サムネイル画像のURL | ○ |
| createdAt | Timestamp | 自動 | 作成日時 | ○ |
| updatedAt | Timestamp | 自動 | 更新日時 | ○ |

### 2.3 機能範囲外（初期バージョン）

以下の機能は初期バージョンには含まれない：

- **共有機能**: ブックマークの共有は不要（プライベートブックマークのみ）
- **インポート/エクスポート**: 不要
- **フォルダ/カテゴリによる整理UI**: データは保存するが、UI機能は将来実装
- **タグフィルタリングUI**: データは保存するが、UI機能は将来実装
- **複数ユーザータイプ/権限管理**: 全ユーザー同権限

## 3. 非機能要件

### 3.1 パフォーマンス要件

**NFR-001: レスポンス時間**
- **要件**: 特定の厳密な要件なし - 標準的なパフォーマンスで問題なし
- **目標**: 
  - ページロード: 5秒以内
  - APIレスポンス: 2秒以内
- **優先度**: 中

**NFR-002: 想定ユーザー負荷**
- **同時ユーザー数**: 1-10ユーザー（個人/小規模チーム）
- **スケーラビリティ**: サーバーレスアーキテクチャにより自動スケール
- **優先度**: 低（小規模想定）

### 3.2 セキュリティ要件

**NFR-003: セキュリティ拡張ルール**
- **適用**: なし（プロトタイプ/実験的プロジェクトとして扱う）
- **理由**: MVPとしての迅速な開発を優先
- **注記**: 本番環境移行時には適用を検討

**NFR-007: モニタリング・ロギング**

- **対象**: Chrome、Firefox、Safari、Edgeの最新バージョンのみ

- **優先度**: 低

### 4.2 技術スタック

#### 4.2.1 フロントエンド
- **フレームワーク**: React
- **言語**: JavaScript / TypeScript
- **ホスティング**: AWS S3 + CloudFront（静的ホスティング）
- **状態管理**: React Context API または Redux Toolkit
- **UIライブラリ**: Material-UI または Tailwind CSS（選択可能）

#### 4.2.2 バックエンド
- **アーキテクチャ**: サーバーレス
- **コンピュート**: AWS Lambda
- **ランタイム**: Node.js (v18以降推奨)
- **API**: AWS API Gateway (REST API)
- **言語**: JavaScript / TypeScript

#### 4.2.3 認証
- **サービス**: AWS Cognito
- **ユーザープール**: Cognitoユーザープール
- **MFAタイプ**: 
  - SMS
  - TOTP (Google Authenticator等)
  - Email
- **認証フロー**: ユーザー名/パスワード + MFA

#### 4.2.4 データベース
- **タイプ**: SQLite (embedded database)
- **配置**: Lambda Layers
- **永続化**: AWS S3（SQLiteファイルを保存）
- **理由**: 
  - 小規模利用でコストがほぼかからない
  - サーバーレスアーキテクチャに適合
  - 1-10ユーザー規模に最適

**データベース設計考慮事項**:
- SQLiteファイルは各Lambda実行時にS3から読み込み
- 書き込み後はS3に保存
- 同時書き込み競合を避けるため、楽観的ロックまたはシーケンシャル処理を実装
- 将来的なスケール時はDynamoDB等への移行パスを考慮

#### 4.2.5 インフラストラクチャ
- **IaC**: AWS CDK または AWS SAM（選択可能）
- **デプロイ**: AWS CLI / CDK Deploy / SAM Deploy
- **モニタリング**: AWS CloudWatch Logs
- **リージョン**: ap-northeast-1（東京）

### 4.3 デプロイメント戦略

**推定コスト**: 月額 $1-5（AWS無料枠活用時）

**コスト内訳**:
- **S3 + CloudFront**: ~$0-1/月（無料枠内）
- **Lambda**: ~$0（無料枠 100万リクエスト/月）
- **API Gateway**: ~$0-1/月（無料枠 100万リクエスト/月 - 最初の12ヶ月）
- **Cognito**: ~$0（無料枠 50,000 MAU）
- **CloudWatch Logs**: ~$0-1/月

### 4.4 開発環境要件

**ローカル開発**:
- Node.js v18以降
- npm または yarn
- AWS CLI v2
- AWS SAM CLI または CDK CLI
- Git

**開発ツール**:
- VSCode または 任意のIDE
- React Developer Tools
- Postman または Thunder Client (API テスト)

## 7. リスクと想定事項

### 7.1 技術リスク

**RISK-001: SQLite同時書き込み競合**
- **説明**: Lambda関数が同時にSQLiteファイルを更新すると競合が発生する可能性
- **影響度**: 中
- **対策**: 
  - 楽観的ロック実装
  - シーケンシャル処理キュー導入
  - DynamoDB Streamsへの将来的な移行パスを設計

**RISK-002: Lambda Cold Start**
- **説明**: Lambda関数のコールドスタート時にレスポンスが遅延する可能性
- **影響度**: 低
- **対策**: 
  - 小規模ユーザー数では許容範囲
  - 必要に応じてProvisioned Concurrency検討

**RISK-003: S3からのSQLiteファイル読み込み遅延**
- **説明**: 各Lambda実行時のS3読み込みがボトルネックになる可能性
- **影響度**: 中
- **対策**:
  - Lambda /tmp ディレクトリでのキャッシング
  - ファイルサイズの監視（大きくなったら分割またはDB移行）

### 7.2 スケーラビリティリスク

**RISK-004: ユーザー数増加時のSQLite制約**
- **説明**: 10ユーザーを超えるとSQLiteの制約が顕著になる可能性
- **影響度**: 高（将来）
- **対策**:
  - ユーザー数を監視
  - 閾値到達時にDynamoDBまたはRDS Auroraへの移行計画を実行
  - マイグレーションスクリプトを事前準備

### 7.3 想定事項

**ASSUMPTION-001**: 1-2週間の開発期間内にMVP機能を完成させる
**ASSUMPTION-002**: AWS無料枠が利用可能であり、月額コストを$5以下に抑えられる
**ASSUMPTION-003**: ユーザー数は初期段階で10名以下に収まる
**ASSUMPTION-004**: 高度な検索/フィルター機能は将来バージョンで実装予定
